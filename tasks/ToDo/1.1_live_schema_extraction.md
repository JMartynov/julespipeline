# Task: 1.1 Live Schema Extraction Engine

### Objective
Build a "Profiler" utility that connects to a live MongoDB URI, utilizes the official MongoDB MCP server to infer the schema, extracts existing indexes, and strictly normalizes the output.

### Utility Design & Modularity
*   **Module Name:** `src/profiler/schema_extractor.py`
*   **Independence:** The utility takes a MongoDB URI and a Database/Collection name as input, and outputs a normalized JSON Schema Draft 2020-12 dictionary along with an array of index definitions.
*   **Configuration:** MCP Server endpoint, transport type, and target URIs must be defined in `config/system.yaml` or `config/mcp.yaml`.

### Implementation Details
*   Connect to the specified MongoDB URI via the official MCP server (v1.10.0).
*   Invoke the `collection-schema` tool.
*   Mandatory: Route the raw MCP output through our existing `src/task2/schema_converter.py` to ensure it is perfectly normalized into standard JSON Schema with `bsonType` hints.
*   Execute `db.collection.getIndexes()` via PyMongo to capture the current state of indexes on the live collection.

### Testing Strategy
1.  **Unit Tests:** Mock the MCP server responses and PyMongo `getIndexes` calls. Ensure the outputs are formatted correctly.
2.  **Comprehensive Acceptance Tests (BDD):** `tests/features/live_extraction.feature`
    *   *Scenario 1 (Happy Path):* Given a valid MongoDB collection with data, when extracted, then a complete JSON Schema 2020-12 and list of indexes are returned.
    *   *Scenario 2 (Edge Case - Empty Collection):* Given an empty collection, when extracted, then a graceful fallback (empty schema or explicit warning) is returned without crashing.
    *   *Scenario 3 (Error Handling - Network):* Given an unreachable MongoDB URI, when extraction is attempted, then a clear `ConnectionTimeoutError` is raised.
    *   *Scenario 4 (Error Handling - MCP Failure):* Given the MCP server returns malformed text instead of JSON, when parsed, then the `schema_converter` safely raises an `MCPParseError`.

### Acceptance Criteria
* [ ] Utility is completely modular and can be instantiated independently.
* [ ] No hardcoded configuration parameters exist in the source code.
* [ ] Unit tests cover 90%+ of the logic.
* [ ] All BDD scenarios in the `.feature` file are written and passing.
