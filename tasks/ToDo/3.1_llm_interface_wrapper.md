# Task: 3.1 LLM Interface Wrapper (Provider-Agnostic)

### Objective
Develop a unified Python wrapper to isolate LLM provider mechanics, strictly enforcing JSON schema outputs for MongoDB optimizations, supporting both LangChain and the Gemini CLI.

### Utility Design & Modularity
*   **Module Name:** `src/llm/wrapper.py`
*   **Independence:** Takes a constructed prompt string/context and returns a strictly typed JSON dictionary.
*   **Explicit LLM Output Schema:** The wrapper must enforce that the LLM returns data perfectly matching this structure:
    ```json
    {
      "action_type": "create_index", // Currently restricted to "create_index"
      "payload": {"status": 1, "created_at": -1}, // The exact index definition
      "justification": "This compound index covers the sort and filters, avoiding an in-memory sort."
    }
    ```
*   **Configuration:** Provider choice (`langchain` vs `gemini_cli`), API keys (via env), model names, temperatures, and the target strict JSON schema must all be defined in `config/llm.yaml`.

### Implementation Details
*   **Engine 1 (LangChain):** Invoke OpenAI/Anthropic models with JSON mode enabled.
*   **Engine 2 (Gemini CLI):** Use `subprocess` to shell out to the authenticated `gemini` CLI, forcing JSON output.
*   **Mock Mode:** If `MOCK_LLM=true`, bypass external calls and return a valid JSON dict from a fixture file.

### Testing Strategy
1.  **Unit Tests:** Test the JSON parsing/cleaning mechanism. Test subprocess calls with mocked stdout.
2.  **Comprehensive Acceptance Tests (BDD):** `tests/features/llm_wrapper.feature`
    *   *Scenario 1 (Happy Path - Mock):* Given MOCK mode, when invoked, then a valid JSON index suggestion is returned instantly without API calls.
    *   *Scenario 2 (Happy Path - Real):* Given `--run-llm-real` is passed, when invoked, then the real API returns a parsed, valid dictionary.
    *   *Scenario 3 (Edge Case - Markdown Cleanup):* Given an LLM returns JSON wrapped in markdown ticks (e.g., ```json ... ```), when parsed, then the ticks are stripped and it parses successfully.
    *   *Scenario 4 (Error Handling - Invalid Schema):* Given the LLM hallucinates keys not allowed by our requested JSON Schema, when validated, then an `LLMSchemaError` is raised (triggering a retry higher up).
    *   *Scenario 5 (Error Handling - Provider Timeout):* Given the LLM API is down, when invoked, then a clear `ProviderTimeoutError` is surfaced.

### Acceptance Criteria
* [ ] Utility is completely modular and can be instantiated independently.
* [ ] No hardcoded configuration parameters exist in the source code.
* [ ] Unit tests cover 90%+ of the logic.
* [ ] All BDD scenarios in the `.feature` file are written and passing.
* [ ] Mock mode is default, fully functioning, and switchable via config/env var.
