# Task: 1.2 Data Profiling & Statistical Sampling

### Objective
Create a statistical profiler utility that analyzes a live MongoDB collection's data distribution and cardinality.

### Utility Design & Modularity
*   **Module Name:** `src/profiler/data_sampler.py`
*   **Independence:** Inputs a PyMongo Collection object and an array of target fields (e.g., indexed fields). Outputs a statistical distribution dictionary (e.g., `{"status": {"active": 0.8, "archived": 0.2}}`).
*   **Configuration:** The maximum sample size (`$sample: { size: 100000 }`) must be configurable in `config/system.yaml`.

### Implementation Details
*   Execute an aggregation pipeline utilizing the configurable `$sample` size.
*   Calculate unique value frequencies (cardinality) for specified fields.

### Testing Strategy
1.  **Unit Tests:** Provide a mocked PyMongo collection with a known distribution. Ensure the profiler accurately calculates the percentages.
2.  **Comprehensive Acceptance Tests (BDD):** `tests/features/statistical_sampling.feature`
    *   *Scenario 1 (Happy Path):* Given a collection with a 90/10 split on a 'status' field, when profiled, then the output dictionary accurately reflects `{"active": 0.9, "archived": 0.1}`.
    *   *Scenario 2 (Edge Case - Massive Collection):* Given a collection with 50M records, when profiled, then the aggregation strictly applies the `$sample` limit and completes in under 5 seconds.
    *   *Scenario 3 (Edge Case - High Cardinality):* Given a field with 100% unique values (like a UUID), when profiled, then the profiler identifies it as highly unique and caps the returned distribution list to top N (configurable) to save memory.
    *   *Scenario 4 (Error Handling):* Given a requested field that does not exist in the collection, when profiled, then it gracefully returns a null distribution.

### Acceptance Criteria
* [ ] Utility is completely modular and can be instantiated independently.
* [ ] No hardcoded configuration parameters exist in the source code.
* [ ] Unit tests cover 90%+ of the logic.
* [ ] All BDD scenarios in the `.feature` file are written and passing.
