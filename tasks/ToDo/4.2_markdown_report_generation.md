# Task: 4.2 Rich Markdown Report Generation

### Objective
Build a templating engine that takes the ranked candidates and generates a human-readable, persistent Markdown report detailing empirical proof and actionable deployment steps.

### Utility Design & Modularity
*   **Module Name:** `src/reporting/generator.py`
*   **Independence:** Takes the Ranked Candidates object and outputs a formatted Markdown string, saving it to disk.
*   **Configuration:** Output directory (`output/reports/`), file naming, and Markdown templates (Jinja2) in `config/reporting.yaml`.

### Implementation Details
*   Generate Executive Summary, Before & After tables, LLM justification, and actionable `mongosh` code block.

### Testing Strategy
1.  **Unit Tests:** Provide mock ranked data. Assert string output does not raise templating errors.
2.  **Comprehensive Acceptance Tests (BDD):** `tests/features/report_generation.feature`
    *   *Scenario 1 (Happy Path):* Given a winning candidate, when generated, then a `.md` file is created on disk containing the exact `db.collection.createIndex()` syntax.
    *   *Scenario 2 (Edge Case - Missing Justification):* Given a candidate where the LLM omitted the justification text, when generated, then the template gracefully inserts a default "No justification provided" string instead of failing.
    *   *Scenario 3 (Error Handling - Read-Only Filesystem):* Given the target output directory is read-only or doesn't exist, when saving, then the generator creates the directory or falls back to standard out (`stdout`) with a warning.

### Acceptance Criteria
* [ ] Utility is completely modular and can be instantiated independently.
* [ ] No hardcoded configuration parameters exist in the source code.
* [ ] Unit tests cover 90%+ of the logic.
* [ ] All BDD scenarios in the `.feature` file are written and passing.
