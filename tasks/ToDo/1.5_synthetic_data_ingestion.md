# Task: 1.5 Synthetic Data Ingestion

### Objective
Provide a highly efficient utility to bulk-insert the generated synthetic data into the isolated Docker testing environment, ensuring the target database is ready for benchmarking.

### Utility Design & Modularity
*   **Module Name:** `src/container/data_ingester.py`
*   **Independence:** Takes an Iterable/Generator of synthetic documents (from Task 1.4) and a PyMongo Collection object pointing *strictly* to the isolated test container.
*   **Configuration:** Batch size for `insert_many()` (e.g., `batch_size: 5000`) must be defined in `config/system.yaml` to optimize memory vs. network speed.

### Implementation Details
*   Iterate through the generated data stream.
*   Accumulate documents up to the configured `batch_size`.
*   Execute `insert_many(batch, ordered=False)` to maximize insertion speed.
*   **Safety Constraint:** The utility must verify that the target URI matches the local Testcontainer connection string and aggressively reject the operation if it points to a production/external URI.

### Testing Strategy
1.  **Unit Tests:** Mock the PyMongo collection. Feed a generator of 12,000 documents and assert that `insert_many` is called exactly 3 times (assuming a batch size of 5000).
2.  **Comprehensive Acceptance Tests (BDD):** `tests/features/data_ingestion.feature`
    *   *Scenario 1 (Happy Path):* Given 10k generated documents and an empty test collection, when ingested, then exactly 10k documents exist in the collection.
    *   *Scenario 2 (Safety Lock):* Given the target URI is identical to the configured Live Source URI, when ingestion is attempted, then a `SecurityError` is raised preventing production corruption.

### Acceptance Criteria
* [ ] Utility is completely modular and can be instantiated independently.
* [ ] No hardcoded configuration parameters exist in the source code.
* [ ] Unit tests cover 90%+ of the logic.
* [ ] All BDD scenarios in the `.feature` file are written and passing.
