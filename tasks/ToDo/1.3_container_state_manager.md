# Task: 1.3 Container & State Manager

### Objective
Decouple the isolated MongoDB testing environment from Pytest so it can be dynamically spun up, managed, and torn down by the production CLI orchestrator.

### Utility Design & Modularity
*   **Module Name:** `src/container/manager.py`
*   **Independence:** Exposes `start_environment()` and `stop_environment()`. Outputs the connection strings (MongoDB URI, MCP Server URL).
*   **Configuration:** Docker images (e.g., `mongo:6.0`, `mongodb-mcp-server:1.10.0`) and internal ports are defined in `config/docker.yaml`.

### Implementation Details
*   Wrap the `testcontainers` library logic into a dedicated Python context manager or class.
*   Ensure the MongoDB container and MCP container can communicate on a shared bridge network.
*   Provide a health-check method to guarantee readiness before returning the URI.

### Testing Strategy
1.  **Unit Tests:** Mock Docker socket responses to ensure the manager attempts to pull/start correct images.
2.  **Comprehensive Acceptance Tests (BDD):** `tests/features/container_manager.feature`
    *   *Scenario 1 (Happy Path):* Given a start request, when executed, then the MongoDB and MCP containers start, and valid local URIs are returned.
    *   *Scenario 2 (Cleanup):* Given a running environment, when stop is requested, then the containers are completely removed from the Docker daemon.
    *   *Scenario 3 (Error Handling - Port Conflict):* Given port 27017 is already in use locally, when started, then `testcontainers` dynamically allocates a free ephemeral port and returns the correct new URI.
    *   *Scenario 4 (Error Handling - Engine Down):* Given Docker is not running, when started, then a clear `DockerEngineUnavailableError` is surfaced.

### Acceptance Criteria
* [ ] Utility is completely modular and can be instantiated independently.
* [ ] No hardcoded configuration parameters exist in the source code.
* [ ] Unit tests cover 90%+ of the logic.
* [ ] All BDD scenarios in the `.feature` file are written and passing.
