# Task: 5.1 CLI Orchestrator (The Main Entrypoint)

### Objective
Build the top-level Command Line Interface (CLI) that wires all independent utilities together into a cohesive, user-facing application flow.

### Utility Design & Modularity
*   **Module Name:** `src/cli.py` or `src/main.py`
*   **Independence:** Uses `argparse` or `click` to accept user arguments and routes them through Phase 1 -> Phase 4 sequentially.
*   **Configuration:** Reads `config/system.yaml` to initialize the sub-modules.

### Implementation Details
*   Provide a command like: `python src/cli.py optimize --uri <URI> --db <DB> --collection <COL> --query slow_query.json`
*   Orchestrate the flow: 
    1. Start Container Manager (1.3)
    2. Extract Schema & Profile Live DB (1.1, 1.2)
    3. Generate Synthetic Data (1.4)
    4. Ingest Query & Get Baseline (2.1, 2.2)
    5. Run Iterative Challenger (3.2)
    6. Rank & Generate Report (4.1, 4.2)
    7. Stop Container Manager (1.3)

### Testing Strategy
1.  **Unit Tests:** Mock all sub-modules and verify the orchestrator calls them in the exact correct sequence.
2.  **Comprehensive Acceptance Tests (BDD):** `tests/features/cli_orchestrator.feature`
    *   *Scenario 1 (Happy Path - E2E MOCK):* Given valid arguments and `MOCK_LLM=true`, when the CLI is executed, then the entire pipeline runs end-to-end and exits with status code 0, producing a report.
    *   *Scenario 2 (Edge Case - Help Menu):* Given the `--help` flag, when executed, then a comprehensive, formatted help menu is printed to stdout.
    *   *Scenario 3 (Error Handling - Missing Args):* Given a missing `--uri` argument, when executed, then the CLI exits with status code 2 and prints a clear validation error.
    *   *Scenario 4 (Resilience - Teardown on Crash):* Given a critical failure in the middle of Phase 3, when the app crashes, then the Container Manager `finally` block guarantees the Docker test containers are still successfully torn down.

### Acceptance Criteria
* [ ] Utility is completely modular and uses standard CLI frameworks (e.g., `argparse`, `click`).
* [ ] No hardcoded configuration parameters exist in the source code.
* [ ] Unit tests cover 90%+ of the orchestration logic.
* [ ] All BDD scenarios in the `.feature` file are written and passing.
