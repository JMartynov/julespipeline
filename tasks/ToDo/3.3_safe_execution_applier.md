# Task: 3.3 Safe Execution Applier

### Objective
Create a secure middle-layer utility that translates the LLM's suggested JSON commands into PyMongo execution calls against the isolated test environment, strictly preventing destructive or unauthorized operations.

### Utility Design & Modularity
*   **Module Name:** `src/optimizer/execution_applier.py`
*   **Independence:** Takes the parsed LLM JSON command and a PyMongo Collection object pointing *strictly* to the isolated test container.
*   **Configuration:** Allowed actions (e.g., `["create_index", "drop_index"]`) should be whitelisted in `config/optimizer.yaml`.

### Implementation Details
*   Parse the `action_type` from the LLM.
*   If `action_type == "create_index"`, translate the `payload` into `pymongo.IndexModel` and execute `collection.create_indexes()`.
*   **Cleanup Mandate:** Provide a `rollback()` method. Since the LLM is iterative, we must aggressively drop any indexes created during Iteration 1 before starting Iteration 2, ensuring each candidate is tested in isolation against the baseline.
*   **Security Lock:** explicitly reject any actions like `drop`, `deleteMany`, `update`, etc.

### Testing Strategy
1.  **Unit Tests:** Pass mocked LLM commands and verify the correct PyMongo calls are generated. Test the rollback logic drops the exact index that was created.
2.  **Comprehensive Acceptance Tests (BDD):** `tests/features/safe_execution.feature`
    *   *Scenario 1 (Happy Path):* Given an LLM command to create an index on `{"status": 1}`, when applied to the test DB, then the index is successfully created.
    *   *Scenario 2 (Rollback):* Given a previously applied index, when rollback is called, then the index is successfully dropped leaving only the original baseline indexes.
    *   *Scenario 3 (Security Rejection):* Given an LLM command hallucinating `action_type: "drop_collection"`, when applied, then an `UnauthorizedActionError` is raised and the DB remains untouched.

### Acceptance Criteria
* [ ] Utility is completely modular and can be instantiated independently.
* [ ] No hardcoded configuration parameters exist in the source code.
* [ ] Unit tests cover 90%+ of the logic.
* [ ] All BDD scenarios in the `.feature` file are written and passing.
