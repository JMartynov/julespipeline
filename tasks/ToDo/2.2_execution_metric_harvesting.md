# Task: 2.2 Execution & Metric Harvesting

### Objective
Create a benchmark runner that executes a parsed query against the isolated Docker container and parses the MongoDB `explain()` output into standard Golden Metrics.

### Utility Design & Modularity
*   **Module Name:** `src/benchmarking/executor.py`
*   **Independence:** Takes a PyMongo Collection object and a normalized Query dictionary. Outputs a standard Metric Dictionary (e.g., `{"executionTimeMillis": 150, "totalDocsExamined": 10000, ...}`).
*   **Configuration:** Timeout thresholds and target `explain` verbosity (e.g., `executionStats`) should be defined in `config/system.yaml`.

### Implementation Details
*   Apply the query to the isolated container.
*   Append `.explain("executionStats")` to the query.
*   Parse the resulting raw JSON to extract: `executionTimeMillis`, `totalDocsExamined`, `totalKeysExamined`.
*   Detect in-memory sorts by scanning the execution stages for `hasSortStage`.

### Testing Strategy
1.  **Unit Tests:** Mock the raw JSON return of an `explain()` command and assert the harvester correctly extracts the target metrics.
2.  **Comprehensive Acceptance Tests (BDD):** `tests/features/metric_harvesting.feature`
    *   *Scenario 1 (Happy Path - COLLSCAN):* Given an unindexed query, when harvested, then `totalDocsExamined` matches the collection count and `executionTimeMillis` is successfully captured.
    *   *Scenario 2 (Happy Path - Indexed):* Given an indexed query, when harvested, then `totalKeysExamined` is > 0 and `totalDocsExamined` drops significantly.
    *   *Scenario 3 (Edge Case - Query Timeout):* Given a query that takes longer than the YAML-configured timeout (e.g., 30s), when executed, then it is terminated and marked with `TimeoutError` in the metrics.
    *   *Scenario 4 (Edge Case - In-Memory Sort):* Given a query requiring a massive sort without an index, when executed, then the `hasSortStage` flag is set to `True` in the output metrics.

### Acceptance Criteria
* [ ] Utility is completely modular and can be instantiated independently.
* [ ] No hardcoded configuration parameters exist in the source code.
* [ ] Unit tests cover 90%+ of the logic.
* [ ] All BDD scenarios in the `.feature` file are written and passing.
