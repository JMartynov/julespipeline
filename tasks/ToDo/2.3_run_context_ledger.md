# Task: 2.3 Run Context & State Ledger

### Objective
Build a definitive state manager that records the Session ID, Baseline Metrics, and all LLM Candidate iterations during a single optimization run.

### Utility Design & Modularity
*   **Module Name:** `src/benchmarking/run_context.py`
*   **Independence:** A standalone singleton or managed object that accepts append operations (save baseline, save candidate) and retrieval operations (get best, get all). 
*   **Configuration:** Storage backend (e.g., `memory`, `sqlite`) and output directory for local DBs must be in `config/system.yaml`.

### Implementation Details
*   Create a data structure/SQLite schema to track `run_id`, `baseline_metrics`, and a list of `candidates` (each holding the suggested index, the explanation, and the resulting metrics).

### Testing Strategy
1.  **Unit Tests:** Validate inserting objects, retrieving objects, and serialization to JSON/SQLite.
2.  **Comprehensive Acceptance Tests (BDD):** `tests/features/run_context.feature`
    *   *Scenario 1 (Happy Path):* Given a new run, when a baseline and 3 candidates are appended, then the context accurately returns all 4 records upon query.
    *   *Scenario 2 (Persistence):* Given SQLite mode is active, when a run finishes, then the data persists to disk and can be queried by a secondary run ID later.
    *   *Scenario 3 (Error Handling - Duplicate ID):* Given an attempt to initialize a session with an existing ID, when executed, then it gracefully increments or raises a `SessionExistsError`.

### Acceptance Criteria
* [ ] Utility is completely modular and can be instantiated independently.
* [ ] No hardcoded configuration parameters exist in the source code.
* [ ] Unit tests cover 90%+ of the logic.
* [ ] All BDD scenarios in the `.feature` file are written and passing.
