# Task: 3.2 The "Iterative Challenger" Loop

### Objective
Build the core orchestrator that converses with the LLM wrapper, executes its suggestions against the test container, evaluates the new metrics, and loops for $N$ iterations.

### Utility Design & Modularity
*   **Module Name:** `src/optimizer/challenger.py`
*   **Independence:** Takes Baseline Metrics, Schema, Query, and LLM Wrapper. Outputs to the Run Context Ledger.
*   **Configuration:** Iterations (`max_iterations`), improvement threshold (`target_latency_reduction_pct: 80`), and prompt templates in `config/optimizer.yaml`.

### Implementation Details
*   Bundle context, invoke LLM, apply the structural change to the test container, and harvest new metrics.
*   Feed delta back to LLM, loop until `max_iterations` or target threshold is met.

### Testing Strategy
1.  **Unit Tests:** Mock the LLM Wrapper and Executor. Verify loop increments and halts correctly.
2.  **Comprehensive Acceptance Tests (BDD):** `tests/features/iterative_loop.feature`
    *   *Scenario 1 (Happy Path - N Iterations):* Given `max_iterations=3` and no candidate meets the 80% threshold, when run, then the loop executes exactly 3 times and saves 3 candidates to the ledger.
    *   *Scenario 2 (Happy Path - Early Exit):* Given a target of 80% reduction, when Iteration 1 achieves a 90% reduction, then the loop halts immediately and skips Iteration 2 and 3.
    *   *Scenario 3 (Error Handling - Bad Syntax from LLM):* Given the LLM suggests an invalid MongoDB index syntax, when execution fails in the test container, then the error is caught, fed *back* to the LLM as part of the next prompt, and the loop continues without crashing the master process.
    *   *Scenario 4 (Error Handling - Max Retries):* Given the LLM fails to provide valid JSON 3 times in a row, when retrying, then the loop gracefully aborts the iteration and continues.

### Acceptance Criteria
* [ ] Utility is completely modular and can be instantiated independently.
* [ ] No hardcoded configuration parameters exist in the source code.
* [ ] Unit tests cover 90%+ of the logic.
* [ ] All BDD scenarios in the `.feature` file are written and passing.
* [ ] Loop runs seamlessly in MOCK mode.
